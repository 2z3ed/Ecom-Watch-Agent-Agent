# P7：老板需求流（B 侧业务服务层）讲解稿（1-2 页）

面向对象：老板 / 面试官  
目标：用“发现 -> 加入 -> 查询 -> 管理”一条线讲清楚 **为什么做项目 B**、**它解决了什么**、**现在能看到什么结果**、**为什么不急着做 A/B 对接**。

---

## 1. 老板怎么提需求（业务语言）

老板/业务同学的核心诉求通常不是“我要一个爬虫”，而是：

- 我想**发现**可能值得关注的竞品商品
- 我想把其中几个**加入监控**
- 我想每天打开就能看到**今天发生了什么变化**，以及**要不要做动作**
- 我想点进某个商品看**详情**
- 我想随时**暂停/恢复/删除**某个监控对象

这些需求的关键词是：**可发现、可选择、可回查、可行动、可管理**。

---

## 2. 系统怎么处理（B 的职责边界）

项目 B 的定位是“业务服务层”，负责把底层采集与分析能力封装成**老板能理解的业务动作**：

### A. 发现（Discovery -> 候选池）

输入：一句话 query（例如“wireless headphone”）  
输出：候选列表（URL + 标题 + domain）并落库留痕（candidate batch / candidate items）

价值：

- discovery 不稳定也没关系，候选池能把“当时搜到什么”固化下来
- 为后续加入监控提供稳定输入（候选项 ID）

### B. 加入监控（候选项/URL -> 正式监控对象）

两种入口：

- 从候选项加入（适合老板“挑选”）
- 从 URL 直接加入（适合运营已经有链接）

加入后立即建立一次最小 baseline：

- 首次采集
- 首快照写入（作为留痕起点）

价值：

- 监控对象不是“发现结果”，是“确认要持续关注的对象”
- baseline 让后续变化检测有可对比的起点

### C. 查询（今日摘要 / 商品详情 / 最近报告）

老板视角查询三件事：

- **今日摘要**：今天有多少在监控？多少发生变化？有没有高优先级？建议动作是什么？
- **商品详情**：当前状态 + 最近变化 + 最新建议
- **最近报告**：快速拿到一条“可执行建议”

价值：

- 不要求老板理解数据库表与内部 task
- 输出更像“业务报表”，可被 A 直接消费

### D. 管理（暂停 / 恢复 / 删除）

最小管理动作：

- 暂停：不再参与后续监控任务
- 恢复：重新参与监控
- 删除：当前实现为软删除/停用（保留留痕，方便回查）

价值：

- 老板需要“随时能控”，而不是只能跑一次 demo

---

## 3. 返回什么结果（演示时你会看到）

建议现场演示用脚本串联一条链：

- `scripts/run_p7_boss_flow_demo.py`

它会按 5 个阶段输出（偏老板视角）：

1. 发现候选：候选数量 + 预览
2. 加入监控：加入方式 + baseline 状态
3. 今日摘要：监控数/变化数/高优先级数 + 建议动作预览
4. 商品详情：当前价格/库存/活动 + 最近变化 + 建议动作
5. 暂停/恢复：状态变化

---

## 4. 为什么先做 B 的业务服务层，再做 A/B 对接

先做 B 的原因：

- A 作为入口层（飞书/交互/编排）需要依赖一个**稳定、可预测**的业务接口面
- 如果没有 B 的“老板语义接口”，A 只能直接耦合到底层采集、DB、diff、LLM 细节，迭代会越来越重
- 先把 B 做成“业务动作集合”，A 才能专注在入口体验（卡片、会话、按钮）与编排

当前不做 A/B 对接的原因：

- 环境/回调约束导致入口层很容易不稳定
- 先把业务能力做扎实，入口层对接风险更低、演示更稳

---

## 5. 一句话总结（面试版）

项目 B 不是爬虫，而是把“采集 -> 留痕 -> 变化 -> 分析 -> 回查”封装成老板能用的 **发现/加入/查询/管理** 四个动作；项目 A 未来只需要做入口编排，不需要理解底层实现。
